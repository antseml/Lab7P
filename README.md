# Лабораторная работа №7 по Программированию

Консольное клиент-серверное приложение, управляющее коллекцией маршрутов, с поддержкой многопоточности, авторизации пользователей и постоянного хранения данных в реляционной СУБД PostgreSQL.

---

## Задание и архитектурные требования

Программа является развитием лабораторной работы №6. Основные изменения коснулись многопоточной обработки запросов, интеграции с базой данных и внедрения системы аутентификации.

### Сетевое взаимодействие
* Обмен данными между клиентом и сервером осуществляется по протоколу UDP.
* Сервер: использует датаграммы (`DatagramSocket` / `DatagramPacket`).
* Клиент: использует сетевой канал (`DatagramChannel`) в неблокирующем режиме.
* Объекты (команды и ответы) передаются исключительно в сериализованном (бинарном) виде. Передача "простых" строк запрещена.
* Клиент корректно обрабатывает временную недоступность сервера.

### База данных и Коллекция
* Хранение коллекции организовано в реляционной СУБД PostgreSQL (хранение в XML/JSON файлах полностью удалено).
* Генерация ID объектов делегирована СУБД (используются sequence или автоинкремент).
* Состояние коллекции в памяти (`In-Memory`) обновляется только при успешном добавлении/изменении объекта в БД.
* Все команды чтения данных (`show`, filter, и т.д.) работают с коллекцией в памяти, обеспечивая максимальное быстродействие.
* Для синхронизации доступа к коллекции в памяти используются потокобезопасные обертки (`java.util.Collections.synchronizedXXX`).

### Авторизация и Разграничение прав
* Реализована полноценная регистрация и авторизация пользователей по логину и паролю.
* Пароли в базе данных хэшируются с использованием алгоритма SHA-512.
* Выполнение команд неавторизованным пользователям запрещено. Логин и пароль передаются в объекте запроса при каждом обращении к серверу.
* Каждый объект коллекции привязан к создавшему его пользователю. Все пользователи могут просматривать коллекцию, но модифицировать (`update`, `remove`) — только свои объекты.

### Многопоточность на Сервере
Сервер обрабатывает запросы конкурентно, используя три уровня изоляции задач:
1.  Чтение запросов: Создание нового потока (`java.lang.Thread`) на каждый запрос.
2.  Обработка запросов: Создание нового потока (`java.lang.Thread`) для бизнес-логики команды.
3.  Отправка ответов: Использование пула потоков ForkJoinPool.

## Структура проекта

Проект разделен на три основных модуля для обеспечения слабой связанности (Low Coupling):

* `common` — общий модуль (shared core). Содержит модели данных (`Route`), объекты запросов (`Request`) и ответов (`Response`), а также утилиты для сериализации.
* `client` — клиентский модуль. Отвечает за интерактивное чтение команд из консоли, валидацию данных, отправку UDP-пакетов и вывод результата.
* `server` — серверный модуль. Отвечает за инициализацию пулов потоков, сетевое взаимодействие, работу с СУБД через JDBC, логирование (Log4J2) и обработку бизнес-логики.
